# 抽出牌类公共房间层，象棋不并入

掼蛋自带约 1200 行房间 / 协议 / 会话代码（`server/guandan-rooms.mjs` + `src/games/guandan/online/`），象棋另有一套独立的 158 行 `server/rooms.mjs`，两者互不共用。加入干瞪眼时若照旧，就是把大厅 / 昵称 / 房间 / 座位 / 重连这套东西**抄第三遍**。因此把它抽成牌类公共房间层，由掼蛋与干瞪眼共用。

## Considered Options

- **各游戏各抄一套（现状延续）**：零重构风险，但第三份重复代码之后，任何一个联机 bug 都要修三处。
- **抽公共层，三个游戏全并入**：最彻底，但象棋是 2 人棋类——没有牌、没有 AI 补空座、胜负判定与重连语义都不同，硬并进来只会让公共层长出「只有象棋会走」的分支，抽象反而被污染。
- **抽公共层，只覆盖牌类（选定）**：掼蛋与干瞪眼共用；象棋那 158 行原样不动。

## Consequences

- 公共层的边界是**座位与转发**：连接、昵称、大厅列表、建房 / 加入 / 观战、座位认领、房主开打、掉线重连、AI 补空座位。它**不解析对局消息的内容**（原句写的是「不解析任何对局消息」，实现时发现站不住，见下方修订一节），游戏侧提供一个适配器与运行器。因此公共层里不会出现任何一个游戏的术语。
- 必须参数化 `minSeats / maxSeats`：掼蛋固定 4 人，干瞪眼 2–5 人且由房主建房时选定目标座位数。
- **掼蛋要能原样跑在这层上，才算抽对了。** 掼蛋已上线且稳定，切换后必须跑通掼蛋联机测试全绿 + 真机多 context 冒烟（建房 / 加入 / 对弈 / 重连），确认零回归。

- **「零回归」有一个前置条件，跳过它这三个字就只是口号。** 现有 `tests/guandan-rooms.test.ts` 断言的是 `reg.rooms`、`reg._dispatch()`、`room.seats[i].graceMisses` 这类**私有名**——抽层后它们必然改名或消失，测试必须重写，而重写后的测试跑绿证明不了任何事（改了尺子又用它量）。因此顺序必须是：
  1. 先给 `RoomRegistry` 加只读快照 API，把该测试文件里所有摸私有的断言改成「读快照 + 读 `client.sent`」，私有调用改成从公开入口驱动；
  2. **在旧实现上跑绿，单独 commit 冻结**——这一版才是零回归基线；
  3. 再抽公共层，同一份测试文件**一个字不改**跑绿。
- 象棋被显式排除。日后若真要并入，需要新开 ADR 推翻本条——不要因为「看起来也是房间」就顺手合并。

## 修订 2026-07-22：「不解析任何对局消息」这句话站不住，改成适配器

实现 #11 时按原句去写，撞上四处冲突——旧实现里这些地方都在拆对局消息：

| 位置 | 干了什么 | 性质 |
|---|---|---|
| 出站分发 | 往 `phase === 'playing'` 的公开态里注入本回合剩余毫秒 | 拆消息，但**通用需求**（倒计时） |
| 出站分发 | 用房间层的真实连接态覆盖公开态里的座位在线字段 | 拆消息，但**通用需求**（头像上的掉线/AI 标记） |
| 出站分发 | 认出 `phase === 'dealResult'` 就自动续局 | 拆消息，且**掼蛋专有** |
| 还贡超时 | 认出 `need-tribute` 就起催办定时 | 拆消息，且**掼蛋专有** |

前两条是任何牌类游戏都要的，后两条是掼蛋自己的节奏（干瞪眼没有进贡，续局节奏也未必相同）。
照原句写「公共层完全不看消息内容」，这四件事全都没地方放。

**修订后的规则（owner 2026-07-22 拍板）**：把方向倒过来——公共层不去拆消息，而是把**它知道的事实**
交给游戏适配器，由适配器自己写进自己的消息；游戏自有的节奏也归适配器决定。

```
适配器（掼蛋 / 干瞪眼各一份）
  minSeats / maxSeats                  座位数范围（掼蛋 4~4，干瞪眼 2~5）
  createRunner(room)                   建对局运行器
  decorate(out, { presence, turnRemainMs })   房间层给事实，游戏自己往消息里写
  autoAdvance(out, runner)             要不要自动往下走 → { delayMs, run } | null
  pendingDecision(out, ctx, runner)    要不要起「谁发呆就替他做」的定时
  clearPendingOn(out)                  这批消息是否意味着不用催了

运行器
  start / setAI / syncSeat / spectatorSync / forceAutoPlay / stepAI
  turnSeat()      当前轮到哪座（不在对局中返回 null）
  progress()      单调递增的行棋计数，用来判「有没有真进展」
  isOver()        整盘是否结束（再来一盘的守卫）
  handleGameMessage(seat, msg)   房间层不认识的消息一律原样交给它
```

于是公共层真的只管座位、转发、回合计时、掉线宽限，加一个游戏不必改它一行。

考虑过但否掉的：**约定一组最小公共字段**（比如公开态一律叫某个名字、座位数组一律叫 `seats`），
让公共层只认这几个。写起来最省事，但等于把两个游戏的公开态形状绑死在房间层里，
加第三个游戏就要在这里动刀——正是本 ADR 想避免的那种耦合。

**零回归是这么证明的**：#4 先把掼蛋房间测试改造成「读只读快照 + 断客户端收到了什么」并在**旧实现**
上跑绿冻结（commit `1fe6e2d`）；本次抽层后，那 47 条测试**文件零改动**跑绿，再加真机多 context 冒烟
（建房 / 加入 / 入座 / 开打 / 观战 / 断线重连）。尺子没动过，量出来的才算数。
