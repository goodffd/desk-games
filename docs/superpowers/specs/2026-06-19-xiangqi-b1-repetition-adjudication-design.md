# 子项目 B1：整本竞赛规则细分裁决（原则化核心）设计

日期：2026-06-19
状态：设计已锁定，待写实现计划
范围：B1 循环重复裁决的细分。扩展现有 `engine/repetition.ts`。纯 engine、无新 UI（结果喂 `Game.status`，复用终局提示）。

## 问题
现状只判「长将 / 粗略长捉 / 否则和」，不分组合。B1 升级为原则化裁决：按「将/捉/闲」分类 + 「长打必变」原则 + 攻击等级层次，覆盖常见组合（一将一捉、一将一闲、长将vs长捉等）。

## 决策（brainstorm 已定）
- 范围：**原则化核心**（非整本棋例）。诚实标注不覆盖官方「棋例」全部冷僻例位。
- 纯 engine，可穷举单测。

## 架构

### 1. 每步分类（扩展 `PlyInfo`）
每步循环着法归为 **将 / 捉 / 闲**：
- **将（check）**：`gaveCheck`（走后对方被将），已有。
- **捉（chase）**：精炼 `chaseThreat`——非将、本步**新制造**对一枚**无根（无保护）**非将敌子的吃子威胁；排除**兑**（被捉子可等价反吃捉子）。拦/跟不构成吃子威胁→自然归"闲"；献（送吃）非捉。
- **闲（idle）**：其余（含兑/拦/跟/献/纯不变）。
`PlyInfo` 改为携带分类（如 `kind: 'check' | 'chase' | 'idle'`，或保留布尔由 adjudicate 归类——实现时定，二选一即可）。

### 2. 裁决：攻击等级层次（重写 `adjudicateRepetition`）
对每方在一个循环里算"攻击等级"：
- **2 = 长将**：该方每步皆"将"。
- **1 = 长打**：每步皆"打"（将或捉）但非全将。
- **0 = 闲**：含任一"闲"步（即非长打）。
裁决：
- 两方等级**相同** → **和**（双方长将 / 双方长打 / 双方含闲）。
- 等级**不同** → **高等级方必变、判负**（长将2 > 长打1 > 闲0）。

自洽覆盖：
| 场景 | 红级 | 黑级 | 结果 |
|---|---|---|---|
| 单方长将 vs 闲 | 2 | 0 | 长将方负 |
| 单方长捉 vs 闲 | 1 | 0 | 长捉方负 |
| 长将 vs 长捉 | 2 | 1 | 长将方负（将更严重） |
| 一将一捉(全打) vs 闲 | 1 | 0 | 该方负 |
| 一将一闲（含闲） vs 闲 | 0 | 0 | 和 |
| 双方长将 / 双方长捉 | 同 | 同 | 和 |
| 消极循环 | 0 | 0 | 和 |

### 3. 集成
`game.ts` 每步计算精炼 `PlyInfo`（将/捉分类）；三次重复触发、取最近一圈的机制不变；把 cycle 交给重写的 `adjudicateRepetition`。`hasUndefendedCaptureThreat` 精炼为「新制造威胁 + 排除兑」。

## 非目标（B1 内）
- 不覆盖官方「棋例」全部例位（几百个）。
- 不做精确"得子数"分析（用无根/等价反吃的近似）。
- 不细分兑/献/拦/跟为独立类别（统归"闲"用于裁决；够用）。

## 验收标准
- **repetition 单测（穷举，重型）**，每个命名场景各有用例：
  - 长将判负；双方长将和；长捉判负；一将一捉判负；一将一闲和；长将 vs 长捉（长将方负）；双方长捉和；消极循环和。
  - 分类正确性：兑/拦/跟 不被误判为"捉"（→ 闲 → 不误判负）。
  - 至少一盘**强制序列集成测试**（Game 上真实走出某循环→裁决正确）。
- 既有 repetition/game 测试不回归。
- `npm test` + `npm run typecheck` 全绿。

## 实现顺序
1. 重写 `adjudicateRepetition` 的攻击等级层次 + 扩展 `PlyInfo` 分类 + 单测（先用构造的 PlyInfo 序列直测裁决逻辑）。
2. `game.ts` 精炼每步分类（捉：新威胁 + 排除兑）+ 集成测试（强制序列）。

## 已知限制（写入注释）
- 原则化裁决 + 将/捉/闲 三类 + 攻击等级层次；不等于官方整本棋例。
- "捉"用无根 + 排除等价反吃的近似，复杂得子/兑献场景可能与官方裁决有出入。
